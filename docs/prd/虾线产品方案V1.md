# 《虾线产品方案 V1》

## 1. 产品定义

### 1.1 产品名称

**虾线**

### 1.2 产品定位

一个面向团队的 **OpenClaw / 龙虾工作流协同平台**。
它不是普通任务管理工具，而是把“人 + 龙虾 + 流程 + 交付”组织成一条可执行、可追踪、可交接、可复用的工作链。

### 1.3 要解决的问题

当前很多龙虾是“个人使用”状态，存在以下问题：

1. 流程分散：工作过程散落在 Excel、群消息、口头通知、截图、人工催办里。
2. 责任不清：龙虾做了什么、人确认了什么、谁最终负责，经常说不清。
3. 交接困难：中间过程没有结构化沉淀，换人就断。
4. 结果不可复用：同类场景每次都从头做，无法形成标准模板。
5. 人机协作边界模糊：哪些该自动执行，哪些必须人工确认，没有统一框架。

### 1.4 核心理念

**龙虾负责执行，人负责结果。**
虾线的核心价值不是“展示任务”，而是把一条原本靠人肉推进的业务链，沉淀为：

* 可编排
* 可执行
* 可协同
* 可留痕
* 可交付
* 可复盘

---

## 2. 产品目标

## 2.1 V1 目标

先打透两个典型场景：

1. **续报后甩合班**

   * 从“人工计算 + 群通知 + 人工催意愿 + 人工测算 + 小工具执行”变成标准化流程。
2. **班主任甩班申请**

   * 从“逐层审批 + 线下沟通”变成“协同看板 + 评论留痕”。

## 2.2 成功标准

V1 达到下面几个点，就算站住了：

* 能配置流程模板
* 能发起流程实例
* 能把任务拆成节点在线流转
* 每个节点可绑定责任人和龙虾
* 节点输入输出可沉淀
* 评论、附件、截图、审批意见可留痕
* 最终能产出“可交付结果”

## 3. 用户角色

## 3.1 主要角色

### 1）流程发起人

发起一次业务流程实例的人。
例如：班级组负责人、运营负责人、小组长。

### 2）节点责任人

对某个节点结果负责的人。
负责确认、补材料、驳回、推进。

### 3）龙虾执行体

负责执行自动化动作或 AI 处理任务。
例如：数据拼表、规则筛选、文案生成、外部接口调用。

### 4）审核人

对关键节点进行确认的人。
例如：运营确认、组长确认、中台一审。

### 5）协作者

可以查看、评论、补充材料，但不一定拥有节点流转权限。

### 6）管理员

负责模板配置、角色权限、流程运行规则、审计查看。

## 4. 信息架构

我建议 V1 的信息架构不要做复杂门户，直接围绕 6 个核心对象展开。

## 4.1 核心对象

### 1）工作空间 Workspace

按业务域或团队划分。
例如：

* 班级运营
* 客服运营
* 数据协同
* 中台流程

### 2）流程模板 Template

定义一条标准工作流。
包括：

* 流程名称
* 节点编排
* 节点规则
* 角色配置
* 默认龙虾绑定
* 表单与输出结构

### 3）流程实例 Run

模板的实际运行记录。
包括：

* 发起时间
* 发起人
* 当前状态
* 当前卡点
* 节点执行记录
* 最终交付结果

### 4）节点 Node

流程里的最小执行单元。
包括：

* 节点类型
* 输入
* 输出
* 负责人
* 龙虾
* 状态
* 评论
* 附件
* 审核结果

### 5）人员 Person

系统中的真实责任主体。
强调“对结果负责”。

### 6）龙虾 Agent

系统中的执行主体。
强调“对动作负责”。

## 5. 核心页面设计

* 业务域
* 模板分类
* 状态
* 负责人
* 发起人
* 时间范围

右侧列表字段：

* 实例名称

* 当前阶段
* 当前责任人
* 当前节点

* 发起时间

给管理者看“现在有哪些流程在跑、卡在哪、谁该处理”。

## 5.2 页面二：流程画布页

* 节点名

* 龙虾

这页解决“全链路可视化”。

## 5.3 页面三：节点执行页 / 侧边详情页

### 作用

真正执行一个节点，不只是看状态。

### 页面模块

* 节点说明
* 前置结果引用
* 当前输入表单
* 运行龙虾按钮
* 运行日志
* 输出结果预览
* 提交审核 / 打回 / 补材料 / 完成
* 评论与附件

### 示例动作

对于“续报后甩合班”中的某些节点：

* 数据准备节点：上传或自动获取课程表、班主任表
* 规则判断节点：运行规则引擎筛出符合条件人群
* AI 节点：生成通知模板或汇总结果
* 执行节点：导入班级 ID 并调用甩班工具
* 人工确认节点：确认接班名单、确认截图有效

输入可能来自：


## 7. 状态机设计

## 7.1 流程实例状态机

### 状态定义

* Draft 草稿
* Pending 待启动
* Running 进行中
* Waiting 等待人工确认
* Blocked 阻塞中
* Failed 失败
* Completed 已完成
* Closed 已关闭
* Cancelled 已取消

### 状态流转

* Draft -> Pending
* Pending -> Running
* Running -> Waiting
* Waiting -> Running
* Running -> Blocked
* Blocked -> Running
* Running -> Failed
* Failed -> Running
* Running -> Completed
* Pending / Running / Waiting -> Cancelled
* Completed -> Closed

### 触发条件

* 发起流程
* 节点执行完成
* 某关键节点要求人工确认
* 某节点超时或执行失败
* 人工取消
* 交付确认完成

---

## 7.2 节点状态机

### 状态定义

* NotStarted 未开始
* Ready 待处理
* Running 执行中
* WaitingConfirm 待确认
* WaitingMaterial 待补材料
* Approved 已通过
* Rejected 已驳回
* Skipped 已跳过
* Failed 已失败
* Done 已完成

### 状态流转示例

* NotStarted -> Ready
* Ready -> Running
* Running -> WaitingConfirm
* Running -> WaitingMaterial
* Running -> Failed
* WaitingMaterial -> Running
* WaitingConfirm -> Approved
* WaitingConfirm -> Rejected
* Approved -> Done
* Rejected -> Ready
* Ready / Running -> Skipped

## 8. V1 业务流程落地示例

## 8.1 模板一：续报后甩合班

这个场景在 PDF 中是季度重复工作，当前流程从表格准备、筛选、意愿收集、测算到触达和执行甩班，而且要求当天完成意愿收集和甩班执行，时效性很强。 

### 建议流程节点

1. 生成班级底表
2. 剔除测试课
3. 剔除异常班主任
4. 拼接人服比/小班数/班主任信息
5. 筛选符合合班条件的班主任
6. 发起合班意愿收集
7. 汇总意愿结果
8. 合班测算
9. 生成拟执行名单
10. 班主任上传家长触达截图
11. 审核触达凭证
12. 调用甩班工具执行
13. 输出执行结果与异常报告

### 节点拆分原则

* 1~5：更适合龙虾主做
* 6~10：人机协同
* 11：人工审核
* 12：工具执行
* 13：交付沉淀

---

## 8.2 模板二：班主任甩班申请

这个场景在 PDF 中给出的现状是：小组长发起申请，中台一审，通知班主任触达，小组长确认触达，提供接班名单，运营确认甩班方案并执行。改进目标是协同看板 + 评论区沟通。

### 建议流程节点

1. 小组长发起甩班申请
2. 中台初审
3. 通知班主任触达家长
4. 上传触达记录
5. 小组长确认触达完成
6. 提供接班名单
7. 运营确认甩班方案
8. 执行甩班
9. 输出结论并归档

### 设计重点

这个模板要重点验证：

* 评论区沟通
* 审批不是纯串行卡死
* 信息补充能在线沉淀
* 所有人看到同一个事实面

## 9. 评论协同设计

这是 V1 必做项，不然方案2落不下来。

## 9.1 评论能力

每个流程、每个节点都支持：

* 普通评论
* @人
* @角色
* 上传附件
* 引用节点输出
* 标记待处理
* 标记已解决

## 9.2 评论沉淀规则

评论不能只是聊天，要支持沉淀为：

* 决策记录
* 风险说明
* 驳回原因
* 补充材料要求
* 结论确认

## 9.3 评论和流程的关系

评论不改变流程，但可以触发动作：

* 评论后提醒责任人
* 补充完材料后允许重新提交
* 管理者可以在评论中做结论归档

## 11.2 V1 不做

这些先忍住，不然会拖死：

* 复杂 BPMN 全量能力
* 无限自定义节点 DSL
* 多租户计费体系
* 高级 BI 报表
* 多工作空间数据隔离深度能力
* 超复杂自动重试编排
* 超多外部系统深集成
* 完整移动端
* 节点级 AI 自我编排

---

## 11.3 V1 技术取舍建议

### 流程编排层

先做“半结构化工作流”，不要追求通用引擎。
支持：

* 串行
* 并行
* 条件分支
* 人工挂起
* 驳回重跑

### 龙虾执行层

先做“节点绑定龙虾”的模式，不做动态 agent swarm。
即：

* 一个节点绑定一个默认龙虾
* 支持手动切换龙虾
* 支持重试

### 数据层

先结构化这几类对象：

* template
* run
* node
* comment
* attachment
* deliverable

* 今日待处理节点
* 异常流程提醒
* 即将超时流程

底部：

* 最近运行的流程实例

### 右侧侧栏

* 最近评论
* 最近交付
* 最近异常日志

这样比纯画布更像一个真平台。

---

## 13. V1 版本验收标准

我建议你把验收标准直接写成这 8 条：

1. 能创建并发布一个流程模板
2. 能基于模板发起一次流程实例
3. 能在画布上看到节点流转
4. 每个节点都能绑定责任人和龙虾
5. 节点能沉淀输入、输出、日志、附件
6. 评论区能承载协同沟通
7. 能完成一次完整交付
8. 能用“续报后甩合班”和“班主任甩班申请”两个模板跑通真实业务样例

## 14. 一句话总结

**虾线 V1 的本质，不是做一个“看起来像流程图的管理系统”，而是先做出一条真正能跑通的人机协作交付链。**

它的最小成立条件不是页面多好看，而是这三件事成立：

* 流程能跑
* 责任能落
* 结果能交付
