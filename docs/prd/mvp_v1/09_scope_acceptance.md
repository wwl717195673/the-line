# 09 范围与验收模块

---

## 1. 模块目标

本模块明确 MVP V1 的不做范围、替代方案、开发顺序和总体验收标准。

目标是防止实现过程中范围发散，保证第一版优先跑通“班主任甩班申请”的人机协作闭环。

---

## 2. 前端与后端实现边界

### 2.1 前端页面边界

MVP 必须实现的前端页面：

* 工作台
* 模板列表页
* 模板详情页
* 流程发起页
* 流程实例列表页
* 流程详情页
* 节点详情页
* 人员管理页
* 龙虾管理页
* 交付中心列表页
* 交付结果页

MVP 明确不实现的前端页面：

* 拖拽式模板设计器
* BPMN 画布
* 高级 BI 报表
* 完整通知中心
* 审计日志后台
* 移动端页面
* PDF / Excel 导出配置页

### 2.2 后端功能边界

MVP 必须实现的后端能力：

* 人员和龙虾基础 CRUD
* 固定模板查询
* 基于固定模板发起流程
* 自动生成 9 个实例节点
* 流程实例列表和详情查询
* 节点输入、提交、审核、驳回、补材料、完成、异常
* 龙虾模拟执行
* 评论、附件和节点日志
* 交付物生成、查询和验收

MVP 明确不实现的后端能力：

* 通用 BPMN 引擎
* 模板拖拽编排服务
* 条件分支执行引擎
* 并行节点调度
* 真实 OpenClaw 调用
* 真实甩班工具调用
* 完整通知投递系统
* 完整审计日志查询系统
* 多租户计费系统
* PDF / Excel 导出服务

## 3. MVP 不做范围

### 3.1 不做复杂 BPMN 全量能力

不做内容：不支持 BPMN 标准中的事件、网关、子流程、泳道、补偿、边界事件等完整能力。

MVP 替代方案：只支持固定模板的串行节点推进。

后续触发条件：当至少 3 条真实业务流程都需要复杂分支、并行或回退规则时，再评估是否引入流程引擎。

### 3.2 不做拖拽式模板设计器

不做内容：不提供前台拖拽节点、连线、调整节点顺序、发布模板的页面。

MVP 替代方案：通过后台预置“班主任甩班申请”模板。

后续触发条件：当业务方需要频繁自行调整流程节点时，再做模板设计器。

### 3.3 不做复杂并行和条件分支

不做内容：不支持一个节点完成后同时激活多个节点，也不支持按条件自动选择不同路径。

MVP 替代方案：按 `sort_order` 串行推进。

后续触发条件：当“续报后甩合班”等流程需要不同条件进入不同处理路径时，再补分支能力。

### 3.4 不做真实外部系统深集成

不做内容：不接教务系统、班级系统、人员系统、消息系统、文件系统等外部系统的深度接口。

MVP 替代方案：通过表单、附件和人工回填记录外部结果。

后续触发条件：当人工回填成为效率瓶颈，且流程字段稳定后再做集成。

### 3.5 不做真实甩班工具调用

不做内容：执行甩班节点不直接调用真实甩班工具。

MVP 替代方案：执行人手动回填执行结果和异常说明。

后续触发条件：当执行字段、权限、安全校验和回滚方案明确后，再接真实工具。

### 3.6 不做真实 OpenClaw 执行集成

不做内容：不调用真实 OpenClaw 任务，不做 agent 任务队列、回调、失败重试。

MVP 替代方案：点击“运行龙虾”只生成模拟日志和模拟输出。

后续触发条件：当节点输入输出结构稳定，并明确某类节点适合自动执行时，再接真实 OpenClaw。

### 3.7 不做完整通知中心

不做内容：不做站内信、邮件、短信、企微、飞书、钉钉等完整通知集成。

MVP 替代方案：待办列表和流程详情页展示处理事项。

后续触发条件：当流程在线跑通但用户经常漏处理时，再补通知能力。

### 3.8 不做高级能力

不做内容：

* 多工作空间深度隔离
* 多租户计费体系
* 高级 BI 报表
* 节点自动催办
* 完整审计日志
* 模板版本管理
* 完整移动端
* 节点级 AI 自我编排
* PDF / Excel 导出

MVP 替代方案：

* 单业务空间试用
* 页面在线查看
* 节点执行日志替代完整审计
* 后台预置模板替代版本管理
* 桌面 Web 优先

## 4. 开发顺序建议

### 第一阶段：数据与模板

目标：建立可发起流程的基础。

开发内容：

1. 建立人员、龙虾、模板、模板节点、实例、实例节点基础模型
2. 实现人员管理基础接口和页面
3. 实现龙虾管理基础接口和页面
4. 预置“班主任甩班申请”模板
5. 支持从模板生成流程实例和实例节点

验收结果：

* 能看到固定模板
* 能维护人员和龙虾
* 能基于模板生成 9 个实例节点

### 第二阶段：流程主链路

目标：让流程能从发起推进到完成。

开发内容：

1. 实现流程实例列表
2. 实现流程详情页
3. 实现节点详情页
4. 实现节点状态流转
5. 实现流程串行推进

验收结果：

* 能发起流程
* 能查看流程详情
* 能处理当前节点
* 完成节点后能进入下一个节点

### 第三阶段：协同与留痕

目标：让协同和处理过程可沉淀。

开发内容：

1. 实现评论
2. 实现附件
3. 实现节点日志
4. 实现龙虾模拟执行

验收结果：

* 评论能沉淀在流程或节点下
* 附件能绑定到正确目标
* 关键动作能写入日志
* 运行龙虾能产生模拟输出

### 第四阶段：交付闭环

目标：让流程完成后能形成可验收交付物。

开发内容：

1. 实现交付物生成
2. 实现交付结果页
3. 实现交付验收状态
4. 实现交付中心列表

验收结果：

* 已完成流程能生成交付物
* 交付页能展示最终结果
* 验收人能通过或驳回交付物

---

## 5. 总体验收标准

MVP V1 必须满足：

1. 能看到预置的“班主任甩班申请”模板
2. 能基于该模板发起流程实例
3. 发起后能自动生成 9 个实例节点
4. 能在流程详情页看到整条流程的节点状态
5. 能处理当前节点，并推动流程进入下一个节点
6. 审核节点支持通过、驳回、要求补材料
7. 节点支持输入、输出、评论、附件、日志
8. 节点支持绑定龙虾并产生模拟执行记录
9. 流程完成后能生成交付物
10. 交付页能展示流程摘要、关键结论、关键附件和验收状态
11. 已取消流程不能继续处理节点
12. 已完成流程不能被取消
13. 历史流程不受后续模板调整影响
14. 停用人员和停用龙虾不能被新节点选择，但历史记录仍可展示

---

## 6. MVP 一句话结论

虾线 MVP V1 不是要先做一个万能流程平台，而是先用“班主任甩班申请”跑通一条可执行、可协同、可留痕、可交付的人机协作链路。
