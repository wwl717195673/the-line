# 《虾线 MVP V1 产品方案》

---

## 1. 产品定义

### 1.1 产品名称

**虾线 MVP V1**

### 1.2 产品定位

一个面向团队的 **OpenClaw / 龙虾工作流协同最小闭环产品**。

MVP V1 不追求做成通用流程引擎，也不追求一步到位覆盖全部业务场景。它的目标是先把一条真实业务流程在线跑通，验证下面三件事是否成立：

* 流程能跑
* 责任能落
* 结果能交付

### 1.3 MVP 核心原则

**先跑通固定流程，再抽象平台能力。**

MVP V1 采用“后台预置模板 + 在线发起实例 + 节点协同处理 + 留痕 + 交付归档”的模式，不做复杂拖拽编排和无限自定义能力。

---

## 2. MVP 目标

## 2.1 目标场景

MVP V1 优先选择 **班主任甩班申请** 作为第一条标准流程。

选择原因：

* 外部系统依赖相对少
* 更适合验证评论协同和轻审批
* 能快速跑通发起、审核、补材料、确认、执行、归档的完整链路
* 比“续报后甩合班”更适合作为第一版最小闭环

“续报后甩合班”暂不作为 MVP V1 的完整实现目标，只作为后续验证龙虾自动化和复杂数据处理能力的第二条模板。

## 2.2 成功标准

MVP V1 达到下面标准，即认为产品最小闭环成立：

* 能基于固定模板发起一次“班主任甩班申请”
* 能查看流程实例当前状态、当前节点、当前责任人
* 能在线处理节点，包括提交、通过、驳回、要求补材料、完成
* 每个节点能沉淀输入、输出、评论、附件、日志
* 节点能绑定一个龙虾，并产生一次模拟执行记录
* 协作者围绕同一个节点事实面沟通，不再依赖群消息散落推进
* 流程完成后能生成交付页
* 整条流程能从发起到归档完整跑通

## 3. 用户角色

MVP V1 只保留最小角色集合，不做复杂权限矩阵。

### 3.1 管理员

负责预置模板、维护人员和龙虾基础信息。

MVP 权限：

* 查看全部流程
* 维护固定模板配置
* 维护人员
* 维护龙虾

### 3.2 发起人

发起一次甩班申请流程的人。

MVP 权限：

* 发起流程实例
* 查看自己发起的流程
* 补充流程材料
* 取消未完成流程

### 3.3 节点责任人

对某个节点处理结果负责的人。

MVP 权限：

* 处理节点
* 填写节点输入
* 上传附件
* 添加评论
* 提交确认

### 3.4 审核人

对关键节点进行确认的人。

MVP 权限：

* 审核通过
* 驳回
* 要求补材料
* 添加审核意见

### 3.5 协作者

参与流程讨论和材料补充的人。

MVP 权限：

* 查看流程
* 评论
* 上传补充材料

### 3.6 观察者

只读查看流程进展的人。

MVP 权限：

* 查看流程
* 查看交付结果

## 4. MVP 信息架构

MVP V1 围绕 6 个核心对象展开。

### 4.1 核心对象

### 1）人员 Person

真实责任主体，强调“谁对结果负责”。

### 2）龙虾 Agent

执行主体，MVP 阶段先作为节点绑定对象和模拟执行对象。

### 3）流程模板 Flow Template

固定流程定义。

MVP 阶段不做前台拖拽设计器，只预置“班主任甩班申请”模板。

### 4）流程实例 Flow Run

一次真实发起的业务流程。

包括：

* 实例标题
* 发起人
* 当前状态
* 当前节点
* 当前责任人
* 输入参数
* 输出结果

### 5）实例节点 Flow Run Node

流程里的最小处理单元。

包括：

* 节点名称
* 节点类型
* 责任人
* 审核人
* 绑定龙虾
* 输入
* 输出
* 状态
* 评论
* 附件
* 日志

### 6）交付物 Deliverable

流程完成后的归档结果。

包括：

* 流程摘要
* 关键结论
* 关键附件
* 审核结论
* 节点完成情况

## 6. MVP 页面设计

## 6.1 页面一：工作台

### 作用

让用户第一眼看到自己需要处理什么，以及当前有哪些流程正在推进。

### 页面结构

顶部：

* 搜索框
* 发起流程

中部：

* 我的待办节点
* 进行中流程
* 最近动态

### MVP 不做

* 复杂统计报表
* 高级风险看板
* 通知中心
* 多维度 BI

---

## 6.2 页面二：流程实例列表页

### 作用

查看当前所有流程实例，并快速定位卡点。

### 列表字段

* 实例名称
* 模板名称
* 当前节点
* 当前责任人
* 状态
* 发起人
* 最近更新时间
* 操作

### 基础筛选

* 状态
* 负责人
* 发起人

### 操作项

* 查看流程
* 取消流程

MVP 阶段暂不做催办和导出。

---

## 6.3 页面三：流程详情页

### 作用

查看一条流程的整体链路、当前卡点和节点处理情况。

### 页面结构

左侧：

* 流程基础信息
* 发起人
* 当前状态
* 当前节点
* 当前责任人

中间：

* 简化流程步骤条或纵向时间线
* 展示每个节点状态
* 高亮当前节点

右侧：

* 节点详情
* 节点输入
* 节点输出
* 评论
* 附件
* 日志
* 操作按钮

### MVP 取舍

MVP 阶段不做复杂拖拽画布，不做 BPMN 图形化编排。用步骤条或时间线表达流程状态即可。

---

## 6.4 页面四：节点详情页

### 作用

承载节点处理动作，是 MVP V1 的核心页面。

### 页面模块

* 节点名称
* 节点说明
* 当前责任人
* 审核人
* 绑定龙虾
* 输入表单
* 附件列表
* 输出结果
* 执行日志
* 评论区
* 操作按钮

### 节点操作

* 暂存
* 提交确认
* 审核通过
* 驳回
* 要求补材料
* 标记完成
* 标记异常
* 运行龙虾

### 龙虾执行说明

MVP 阶段先做“模拟执行”：

* 节点可以绑定一个龙虾
* 点击运行后生成执行日志
* 生成结构化输出占位结果
* 不接真实 OpenClaw 或外部工具

---

## 6.5 页面五：交付结果页

### 作用

展示流程最终交付内容，证明流程不是“走完了”，而是“有结果可归档”。

### 页面模块

* 交付标题
* 流程摘要
* 节点完成情况
* 关键结论
* 关键附件
* 审核意见
* 交付状态

### MVP 不做

* PDF 导出
* Excel 导出
* 复杂复盘模板

MVP 阶段只要求页面可查看、链接可访问。

## 7. MVP 固定流程模板

## 7.1 模板名称

**班主任甩班申请**

## 7.2 流程节点

### 1）小组长发起甩班申请

节点类型：人工处理节点

核心输入：

* 申请原因
* 涉及班级
* 当前班主任
* 期望处理时间
* 补充说明

核心输出：

* 申请单

### 2）中台初审

节点类型：审核节点

核心输入：

* 申请单
* 补充材料

核心输出：

* 通过
* 驳回
* 要求补材料

### 3）通知班主任触达家长

节点类型：通知节点

核心输入：

* 申请单
* 触达要求

核心输出：

* 通知记录

MVP 阶段通知只记录结果，不接真实消息系统。

### 4）上传触达记录

节点类型：留痕节点

核心输入：

* 触达截图
* 触达说明

核心输出：

* 触达凭证

### 5）小组长确认触达完成

节点类型：审核节点

核心输入：

* 触达凭证

核心输出：

* 确认通过
* 驳回补充

### 6）提供接班名单

节点类型：人工处理节点

核心输入：

* 接班班主任
* 接班班级
* 承接说明

核心输出：

* 接班名单

### 7）运营确认甩班方案

节点类型：审核节点

核心输入：

* 申请单
* 触达凭证
* 接班名单

核心输出：

* 运营确认意见
* 最终甩班方案

节点类型：执行节点

核心输入：

* 最终甩班方案

核心输出：

* 执行结果
* 异常说明

### 9）输出结论并归档

节点类型：留痕节点

核心输入：

* 执行结果

* 关键附件

* 交付摘要

## 8. 状态机设计

## 8.1 流程实例状态

MVP V1 流程实例状态：

* draft 草稿
* running 进行中
* waiting 等待人工处理
* blocked 阻塞中
* completed 已完成
* cancelled 已取消

### 状态流转

* draft -> running
* running -> waiting
* waiting -> running
* running -> blocked
* blocked -> running
* running -> completed
* draft / running / waiting -> cancelled

---

## 8.2 节点状态

MVP V1 节点状态：

* not_started 未开始
* ready 待处理
* running 执行中
* waiting_confirm 待确认
* waiting_material 待补材料
* rejected 已驳回
* done 已完成
* failed 已失败

### 状态流转

* not_started -> ready
* ready -> running
* running -> waiting_confirm
* running -> waiting_material
* waiting_material -> running
* waiting_confirm -> done
* waiting_confirm -> rejected
* rejected -> ready
* running -> done
* running -> failed
* failed -> ready

## 11. MVP 需求明细

本章节用于把前文 MVP 范围拆成可实现、可验收的需求条目。后续研发、测试、设计均以本章节作为需求细化依据。

### 11.1 需求说明约定

每个需求需要明确以下内容：

* 目标：说明为什么要做这个能力，以及它解决什么业务问题
* 入口：说明用户从哪个页面或动作进入这个能力
* 前置条件：说明执行该能力前必须满足什么条件
* 业务规则：说明系统应该如何处理、如何校验、如何流转
* 数据落点：说明核心数据写入哪些对象或字段
* 验收标准：说明研发完成后如何判断这个需求可用

MVP 阶段优先保证“流程闭环正确”，体验优化和高级配置能力后置。

## 11.2 基础数据需求

### REQ-BASE-001 人员管理

目标：系统需要知道每个流程和节点由谁负责，否则“责任能落”无法成立。

入口：资源中心 - 人员管理。

前置条件：管理员已登录系统。

业务规则：

* 管理员可以创建人员
* 管理员可以编辑人员基础信息
* 管理员可以停用人员
* 停用人员不能再被分配为新流程节点责任人

目标：发起人需要在流程还未完成时取消错误发起或不再需要推进的流程。

* 当前用户是发起人或管理员

* 取消后 `flow_run.current_status` 更新为 `cancelled`
* 取消后所有未完成节点不再允许处理
* 取消动作写入节点日志或流程日志
* MVP 阶段取消后不支持恢复

### REQ-RUN-005 节点推进

目标：当当前节点完成后，流程需要自动进入下一个节点。

入口：节点详情 - 标记完成；节点详情 - 审核通过。

前置条件：

* 当前节点状态满足完成条件
* 当前流程状态为 `running` 或 `waiting`

业务规则：

* 当前节点完成后，系统查找下一个 `sort_order` 节点
* 如果存在下一个节点，下一个节点状态变为 `ready`
* 如果存在下一个节点，`flow_run.current_node_code` 更新为下一个节点编码
* 如果不存在下一个节点，流程状态变为 `completed`
* 如果流程完成，系统允许生成交付物
* MVP 阶段只支持串行推进

流程状态同步规则：

* 节点进入 `waiting_confirm`、`waiting_material` 或 `rejected` 时，流程状态为 `waiting`
* 节点标记异常时，流程状态为 `blocked`
* 节点完成并激活下一个节点时，流程状态为 `running`
* 最后一个节点完成时，流程状态为 `completed`

数据落点：

* `flow_run.current_node_code`
* `flow_run.current_status`
* `flow_run.completed_at`
* `flow_run_node.status`
* `flow_run_node.completed_at`

验收标准：

* 完成第 1 个节点后，第 2 个节点自动变为待处理
* 当前节点字段随流程推进更新
* 完成第 9 个节点后流程状态变为已完成

* 用户可以在节点下发表评论
* 评论内容不能为空
* 评论按创建时间正序或倒序展示，页面规则需要保持一致
* 评论可以标记为已解决
* MVP 阶段 `@人` 先作为文本保存，不做真实通知
* MVP 阶段不做评论编辑和删除

数据落点：

* `comment.target_type`
* `comment.target_id`
* `comment.author_person_id`
* `comment.content`
* `comment.is_resolved`

验收标准：

* 评论发布后刷新页面仍可见
* 评论能正确归属到对应节点
* 标记已解决后状态能展示

### REQ-COMMENT-002 流程评论

目标：支持对整条流程进行全局说明或结论沉淀。

入口：流程详情 - 流程评论区。

前置条件：用户对流程有查看权限。

业务规则：

* 流程评论绑定到 `flow_run`
* 节点评论绑定到 `flow_run_node`
* 流程评论不改变流程状态
* 重要结论可以后续被人工复制到交付摘要

数据落点：

* `comment.target_type = flow_run`
* `comment.target_id = flow_run.id`

验收标准：

* 流程评论不会出现在某个具体节点评论区
* 节点评论不会混入流程全局评论区

### REQ-ATT-001 附件上传

目标：节点材料、触达截图、接班名单等凭证需要在线沉淀。

入口：节点详情 - 上传附件；评论区 - 添加附件；交付页 - 绑定附件。

前置条件：用户对目标对象有上传权限。

业务规则：

* 附件必须绑定到具体目标对象
* 支持绑定到流程、节点、评论、交付物
* 附件记录保存文件名、文件地址、大小、类型、上传人
* MVP 阶段不做复杂预览和在线编辑
* MVP 阶段不做病毒扫描和复杂文件权限隔离

数据落点：

* `attachment.target_type`
* `attachment.target_id`
* `attachment.file_name`
* `attachment.file_url`
* `attachment.file_size`
* `attachment.file_type`
* `attachment.uploaded_by`

验收标准：

* 上传后能在对应节点或评论下看到附件
* 附件不会串到其他节点
* 刷新页面后附件仍可见

### REQ-LOG-001 节点日志

目标：关键动作需要留痕，后续复盘时能知道谁在什么时候做了什么。

入口：系统自动写入；节点详情 - 日志区查看。

前置条件：用户触发关键动作。

业务规则：

* 创建流程时写日志
* 节点暂存时写日志
* 节点提交时写日志
* 审核通过时写日志
* 驳回时写日志
* 要求补材料时写日志
* 上传附件时写日志
* 运行龙虾时写日志
* 标记异常时写日志
* 日志不可编辑和删除

数据落点：

* `flow_run_node_log.run_id`
* `flow_run_node_log.run_node_id`
* `flow_run_node_log.log_type`
* `flow_run_node_log.operator_type`
* `flow_run_node_log.operator_id`
* `flow_run_node_log.content`
* `flow_run_node_log.extra_json`

验收标准：

* 每个关键动作都能在节点日志中看到
* 日志能显示操作者、动作内容和时间
* 龙虾模拟执行日志能和人工操作日志区分

## 11.7 交付需求

### REQ-DELIVER-001 生成交付物

目标：流程完成后需要产出可归档、可验收的结果，而不是只显示节点都完成了。

入口：流程详情 - 生成交付物；最后一个节点完成后系统提示生成。

前置条件：

* 流程状态为 `completed`
* 当前用户是发起人、管理员或最终节点责任人

业务规则：

* 交付标题默认使用流程标题加“交付结果”
* 交付摘要可以由用户填写
* 系统汇总节点完成情况
* 系统汇总关键附件
* 系统汇总最终执行结果和异常说明
* 对用户选择的关键附件复制附件元信息，创建 `target_type = deliverable`、`target_id = deliverable.id` 的附件绑定
* 交付物初始验收状态为 `pending`
* MVP 阶段不自动生成 PDF 或 Excel

数据落点：

* `deliverable.run_id`
* `deliverable.title`
* `deliverable.summary`
* `deliverable.result_json`
* `deliverable.review_status`

验收标准：

* 已完成流程能生成交付物
* 未完成流程不能生成交付物
* 交付页能展示流程摘要、节点完成情况、关键附件和最终结论

### REQ-DELIVER-002 验收交付物

目标：交付结果需要有明确验收结论，避免流程完成但结果无人确认。

入口：交付页 - 验收通过；交付页 - 验收驳回。

前置条件：

* 交付物已生成
* 当前用户是验收人或管理员

业务规则：

* 验收通过后 `review_status` 变为 `approved`
* 验收驳回后 `review_status` 变为 `rejected`
* 验收时可以填写验收意见
* 验收时间写入 `reviewed_at`
* MVP 阶段验收驳回不自动重开流程

数据落点：

* `deliverable.review_status`
* `deliverable.reviewer_person_id`
* `deliverable.reviewed_at`
* `deliverable.result_json`

验收标准：

* 交付页能看到验收状态
* 验收操作后状态立即更新
* 验收意见能在交付详情中看到

## 12. MVP 不做范围

MVP V1 明确不做以下内容。本章节用于防止研发过程中范围发散。

### 12.1 不做复杂 BPMN 全量能力

不做内容：不支持 BPMN 标准中的事件、网关、子流程、泳道、补偿、边界事件等完整能力。

MVP 替代方案：只支持固定模板的串行节点推进。

后续触发条件：当至少 3 条真实业务流程都需要复杂分支、并行或回退规则时，再评估是否引入流程引擎。

### 12.2 不做拖拽式模板设计器

不做内容：不提供前台拖拽节点、连线、调整节点顺序、发布模板的页面。

MVP 替代方案：通过后台预置“班主任甩班申请”模板。

后续触发条件：当业务方需要频繁自行调整流程节点时，再做模板设计器。

### 12.3 不做无限自定义节点 DSL

不做内容：不提供自定义表达式、脚本节点、复杂条件配置、动态执行规则语言。

MVP 替代方案：节点类型固定为人工、审核、通知、执行、留痕。

后续触发条件：当固定节点类型无法覆盖核心业务流程时，再抽象节点 DSL。

### 12.4 不做复杂并行和条件分支

不做内容：不支持一个节点完成后同时激活多个节点，也不支持按条件自动选择不同路径。

MVP 替代方案：按 `sort_order` 串行推进。

后续触发条件：当“续报后甩合班”等流程需要不同条件进入不同处理路径时，再补分支能力。

### 12.5 不做多工作空间深度隔离

不做内容：不实现完整 workspace 级数据隔离、成员隔离、权限隔离和跨空间切换。

MVP 替代方案：先按单业务空间运行，必要时字段预留 `workspace_id`，但不实现完整隔离逻辑。

后续触发条件：当多个团队同时接入且存在数据隔离诉求时，再补工作空间能力。

### 12.6 不做多租户计费体系

不做内容：不做租户、套餐、用量统计、计费、欠费限制、账单。

MVP 替代方案：内部试用模式，不涉及商业化计费。

后续触发条件：当产品进入对外服务或商业化阶段时再设计。

### 12.7 不做高级 BI 报表

不做内容：不做趋势图、效率分析、节点耗时分布、人员绩效统计、异常率分析。

MVP 替代方案：只在列表和详情页展示流程状态、当前节点、当前责任人。

后续触发条件：当流程数量足够多，管理者需要看整体效率时再做报表。

### 12.8 不做完整通知中心

不做内容：不做站内信、邮件、短信、企微、飞书、钉钉等完整通知集成。

MVP 替代方案：待办列表和流程详情页展示处理事项。

后续触发条件：当流程在线跑通但用户经常漏处理时，再补通知能力。

### 12.9 不做节点自动催办

不做内容：不做 SLA 倒计时、自动催办、超时升级、重复提醒。

MVP 替代方案：流程列表展示当前责任人，人工查看和推进。

后续触发条件：当真实流程中出现明显超时和催办成本时，再做自动催办。

### 12.10 不做真实外部系统深集成

不做内容：不接教务系统、班级系统、人员系统、消息系统、文件系统等外部系统的深度接口。

MVP 替代方案：通过表单、附件和人工回填记录外部结果。

后续触发条件：当人工回填成为效率瓶颈，且流程字段稳定后再做集成。

### 12.11 不做真实甩班工具调用

不做内容：执行甩班节点不直接调用真实甩班工具。

MVP 替代方案：执行人手动回填执行结果和异常说明。

后续触发条件：当执行字段、权限、安全校验和回滚方案明确后，再接真实工具。

### 12.12 不做真实 OpenClaw 执行集成

不做内容：不调用真实 OpenClaw 任务，不做 agent 任务队列、回调、失败重试。

MVP 替代方案：点击“运行龙虾”只生成模拟日志和模拟输出。

后续触发条件：当节点输入输出结构稳定，并明确某类节点适合自动执行时，再接真实 OpenClaw。

### 12.13 不做完整审计日志

不做内容：不做全对象 before / after 级审计，不做审计查询后台。

MVP 替代方案：只做节点执行日志，记录关键动作。

后续触发条件：当涉及合规、追责或管理员审计需求时，再补完整审计日志。

### 12.14 不做模板版本管理

不做内容：不做模板版本列表、版本对比、版本回滚、按版本复制。

MVP 替代方案：`flow_run.template_version` 先保留字段，模板变更通过后台控制。

后续触发条件：当模板开始频繁迭代且历史实例需要按版本回溯时，再实现版本管理。

### 12.15 不做完整移动端

不做内容：不做独立 App、小程序、移动端适配页面。

MVP 替代方案：优先保证桌面 Web 可用。

后续触发条件：当节点责任人大量在移动端处理评论、附件和审核时，再做移动端。

### 12.16 不做节点级 AI 自我编排

不做内容：不让 AI 自动决定下一步流程、不自动新增节点、不自动改派责任人。

MVP 替代方案：流程推进由固定模板和人工操作控制。

后续触发条件：当流程规则稳定且需要降低人工编排成本时，再评估 AI 编排。

### 12.17 不做 PDF / Excel 导出

不做内容：不实现交付物 PDF 导出、Excel 导出、批量下载。

MVP 替代方案：交付页在线查看，附件原文件可访问。

后续触发条件：当业务验收或归档必须离线流转时，再补导出能力。

## 13. 开发顺序建议

### 第一阶段：数据与模板

1. 建立人员、龙虾、模板、模板节点、实例、实例节点基础模型
2. 预置“班主任甩班申请”模板
3. 支持从模板生成流程实例和实例节点

### 第二阶段：流程主链路

1. 实现流程实例列表
2. 实现流程详情页
3. 实现节点详情页
4. 实现节点状态流转

### 第三阶段：协同与留痕

1. 实现评论
2. 实现附件
3. 实现节点日志
4. 实现龙虾模拟执行

### 第四阶段：交付闭环

1. 实现交付物生成
2. 实现交付结果页
3. 实现交付验收状态

## 14. MVP 验收标准

MVP V1 的验收标准如下：

1. 能看到预置的“班主任甩班申请”模板
2. 能基于该模板发起流程实例
3. 发起后能自动生成 9 个实例节点
4. 能在流程详情页看到整条流程的节点状态
5. 能处理当前节点，并推动流程进入下一个节点
6. 审核节点支持通过、驳回、要求补材料
7. 节点支持输入、输出、评论、附件、日志
8. 节点支持绑定龙虾并产生模拟执行记录
9. 流程完成后能生成交付物
10. 交付页能展示流程摘要、关键结论、关键附件和验收状态

---

## 15. 一句话结论

**虾线 MVP V1 不是要先做一个万能流程平台，而是先用“班主任甩班申请”跑通一条可执行、可协同、可留痕、可交付的人机协作链路。**
