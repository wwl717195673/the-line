# 《虾线龙虾集成产品方案》

## 1. 方案目标

本方案用于定义“虾线”下一阶段的核心产品方向：

* 让流程发起人可以通过自然语言描述业务目标，由龙虾生成可确认的流程草案
* 让龙虾在流程流转到自动节点时被真实触发执行任务
* 让龙虾在执行结束后，通过标准回执协议把结果回传给虾线，由虾线决定是否推进流程
* 让“龙虾负责执行，人负责结果”成为产品结构、交互和数据模型中的核心原则

本方案默认先忽略复杂权限体系，优先打通“编排 -> 执行 -> 回执 -> 责任确认 -> 交付”的主链路。

---

## 2. 产品定义

### 2.1 产品定位

虾线不是单纯的工作流系统，也不是单纯的 Agent 运行器。

它是一个把 AI 执行能力和人工责任机制绑定在一起的结果型流程系统。

### 2.2 核心理念

**龙虾负责执行，人负责结果。**

这句话在产品中需要被拆成三层：

1. 龙虾负责完成具体动作
2. 人负责审核、签收、兜底和异常处理
3. 虾线负责把动作、责任和结果串成一条可追踪的流程链

### 2.3 与现有 V1 的关系

现有 V1 更偏向“固定模板 + 人工推进 + 模拟龙虾执行”。

龙虾集成方案是在此基础上扩展出两类新能力：

* **龙虾辅助编排能力**
  * 通过自然语言生成流程草案，而不是只从固定模板发起
* **龙虾真实执行能力**
  * 自动节点流转到龙虾时，能够触发真实执行并自动回执

---

## 3. 核心业务主张

### 3.1 产品要解决的问题

当前业务团队使用龙虾时，往往存在以下断裂：

* 龙虾能执行，但执行过程不在统一流程里
* 人知道结果要负责，但不知道龙虾到底做了什么
* 业务目标可以口头描述，但无法快速变成结构化流程
* 龙虾执行完成后，没有统一系统承接结果、责任人和后续流转

虾线要解决的不是“如何让龙虾更强”，而是：

* 如何让龙虾的生产力进入业务流程
* 如何让人能接住龙虾的结果并对结果负责
* 如何让这一过程可追踪、可复盘、可交付

### 3.2 产品核心价值

虾线的核心价值可以概括为三句话：

* 把自然语言需求变成结构化流程草案
* 把龙虾执行结果变成标准化节点回执
* 把责任确认和最终交付沉淀为可复用流程资产

---

## 4. 目标用户与典型场景

### 4.1 目标用户

1. 流程发起人
   * 提出业务目标，希望快速组织一条人机协作流程
2. 节点责任人
   * 需要确认、审核、补材料、签收结果
3. 龙虾执行体
   * 负责查询、加工、批量操作、导出等自动化执行
4. 最终签收人
   * 对最终结果负责，确认流程交付达标
5. 管理者
   * 关注流程是否顺畅、责任是否明确、结果是否可追溯

### 4.2 典型场景

#### 场景 A：自然语言生成视频绑定流程

发起人对自己的龙虾说：

* 帮我创建一个视频绑定的工作流程
* 第一个自动节点收集距离开课不足 2 天的课程场次数据
* 第二个节点由我审核确认数据
* 第三个节点由龙虾执行录播课资源绑定
* 第四个节点导出绑定情况作为结果交付
* 第五个节点由 X 部门 X 同事确认最终结果

期望结果不是直接上线流程，而是生成一个**流程草案**，由人确认后创建。

#### 场景 B：自动节点真实执行与自动流转

流程进入龙虾节点后：

1. 虾线触发对应龙虾执行任务
2. 龙虾完成查询、绑定或导出动作
3. 龙虾向虾线回传标准化执行回执
4. 虾线根据回执判断：
   * 自动完成
   * 待人工确认
   * 异常升级
5. 流程流转至下一个节点

## 5. 目标产品形态

### 5.1 总体结构

龙虾集成后的虾线由 4 个关键层组成：

1. **需求编排层**
   * 用户用自然语言描述目标
   * 龙虾输出流程草案
2. **流程承接层**
   * 虾线负责保存模板/实例、节点状态、责任人和交付物
3. **任务执行层**
   * 自动节点触发龙虾执行真实任务
4. **责任闭环层**
   * 人工审核、签收、异常处理、交付留痕

### 5.2 页面级产品变化

相较现有 V1，页面上需要新增或强化：

* 发起入口支持“让龙虾生成流程草案”
* 流程草案确认页
* 自动节点执行态展示
* 节点执行结果结构化展示
* 节点责任人和执行主体双视角展示
* 异常节点升级和人工接管入口

## 6. 核心对象模型

### 6.1 保留现有核心对象

现有对象仍然保留：

* `Template`
* `Run`
* `RunNode`
* `Person`
* `Agent`
* `Comment`
* `Attachment`
* `Deliverable`

### 6.2 新增关键对象

建议新增以下对象：

#### 1）流程草案 `FlowDraft`

用于承接自然语言生成的流程草案。

建议字段：

* `id`
* `title`
* `description`
* `source_prompt`
* `creator_person_id`
* `planner_agent_id`
* `status`
  * `draft`
  * `confirmed`
  * `discarded`
* `structured_plan_json`
* `created_at`
* `confirmed_at`

#### 2）节点执行任务 `AgentTask`

用于记录虾线向龙虾发起的一次真实执行请求。

建议字段：

* `id`
* `run_id`
* `run_node_id`
* `agent_id`
* `task_type`
* `input_json`
* `status`
  * `queued`
  * `running`
  * `completed`
  * `needs_review`
  * `failed`
  * `cancelled`
* `started_at`
* `finished_at`
* `error_message`
* `result_json`
* `artifacts_json`

#### 3）节点执行回执 `AgentTaskReceipt`

用于保存龙虾回传的原始回执，便于审计和回放。

建议字段：

* `id`
* `agent_task_id`
* `run_id`
* `run_node_id`
* `agent_id`
* `receipt_status`
* `payload_json`
* `received_at`

### 6.3 节点上必须显式展示的两个主体

每个节点都建议显式区分：

* **执行主体**
  * 谁执行动作
  * 可能是人，也可能是龙虾
* **结果责任人**
  * 谁对节点结果签字负责

这两个主体不能混成一个字段，否则产品的核心价值会丢失。

## 7. 节点类型设计

### 7.1 节点类型

建议将节点类型统一为以下几类：

1. 人工输入节点 `human_input`
   * 人填写信息、补材料、发起申请
2. 人工审核节点 `human_review`
   * 人确认龙虾或上一步的结果
3. 自动执行节点 `agent_execute`
   * 龙虾查询、处理、批量操作
4. 自动导出节点 `agent_export`
   * 龙虾生成报表、导出结果
5. 最终签收节点 `human_acceptance`
   * 业务最终确认结果

### 7.2 节点状态

在现有状态基础上，建议把自动节点的关键状态明确化：

* `not_started`
* `ready`
* `running`
* `waiting_confirm`
* `waiting_material`
* `blocked`
* `done`
* `failed`
* `cancelled`

其中对龙虾节点最关键的是：

* `running`
  * 龙虾已被触发，正在执行
* `waiting_confirm`
  * 龙虾已产出结果，但需要人工确认后才能继续
* `blocked`
  * 龙虾无法完成，需要人工接管

### 7.3 节点必须具备的执行契约

所有龙虾节点都必须有明确契约，不允许只有模糊描述。

建议每个自动节点至少包含：

* 节点目标
* 输入字段
* 可用数据源
* 可调用工具/系统
* 输出结构
* 完成条件
* 失败条件
* 异常升级规则

例如“收集距离开课不足 2 天的课程场次数据”节点，其输出不应只是文本，而应是结构化结果：

* 场次数量
* 场次 ID 列表
* 开课时间
* 当前视频绑定状态
* 缺失原因

## 8. 龙虾辅助编排方案

### 8.1 产品原则

龙虾不直接“创建并上线流程”，而是生成“可确认的流程草案”。

原因：

* 流程本身也是责任对象
* 业务需求天然存在歧义
* 直接上线风险高
* 人必须对流程定义本身负责

### 8.2 编排流程

建议的产品链路：

1. 用户输入自然语言需求
2. 选择或默认绑定一个“编排龙虾”
3. 龙虾输出流程草案
4. 系统展示草案摘要和节点详情
5. 用户调整或确认
6. 系统创建模板或直接创建实例

### 8.3 流程草案输出结构

草案建议至少包含：

* 流程标题
* 流程目标
* 节点列表
* 节点顺序
* 节点类型
* 每个节点的执行主体
* 每个节点的结果责任人
* 每个节点的输入输出
* 继续流转条件
* 失败处理建议
* 最终交付定义

### 8.4 MVP 限制

为保证收敛，MVP 不建议一开始就让龙虾生成任意复杂流程。

建议限制为：

* 串行流程
* 3 到 8 个节点
* 必须至少有一个人工确认节点
* 必须存在最终签收节点
* 自动节点只允许标准任务类型

## 9. 龙虾真实执行方案

### 9.1 触发机制

当流程流转到自动节点时，虾线负责创建一条 `AgentTask`，并向龙虾发送执行请求。

触发条件建议为：

* 节点状态进入 `ready`
* 节点类型为 `agent_execute` 或 `agent_export`
* 节点绑定了可用龙虾
* 节点输入已满足执行条件

### 9.2 执行结果类型

龙虾执行完成后，结果不应只有“成功 / 失败”。

建议支持以下结果：

* `completed`
  * 已按要求完成，可自动进入下一节点
* `needs_review`
  * 已产出结果，但需要人确认
* `failed`
  * 执行失败
* `blocked`
  * 当前缺少关键输入或外部依赖，无法继续

### 9.3 节点流转规则

建议规则如下：

1. 龙虾回执为 `completed`
   * 节点置为 `done`
   * 自动流转到下一节点
2. 龙虾回执为 `needs_review`
   * 节点置为 `waiting_confirm`
   * 等待结果责任人确认
3. 龙虾回执为 `failed` 或 `blocked`
   * 节点置为 `blocked` 或 `failed`
   * 自动创建异常提示并通知责任人处理

## 10. 龙虾回执协议

### 10.1 设计原则

龙虾不是直接修改流程状态，而是向虾线回传一份标准执行回执。

虾线才是唯一的流程状态控制者。

这样可以保证：

* 流程边界清晰
* 执行与编排解耦
* 审计和重放能力更好
* 多种龙虾执行器可以并存

### 10.2 回执字段建议

建议最小回执协议为：

* `run_id`
* `run_node_id`
* `agent_id`
* `agent_task_id`
* `status`
  * `completed`
  * `needs_review`
  * `failed`
  * `blocked`
* `started_at`
* `finished_at`
* `summary`
* `result`
* `artifacts`
* `logs`
* `error_message`

### 10.3 结果字段约束

建议 `result` 使用结构化 JSON，而不是纯文本。

例如：

* 查询节点输出 `records`
* 批量操作节点输出 `success_count`、`failed_count`
* 导出节点输出 `file_url`、`file_name`

### 10.4 回执后的系统动作

虾线收到回执后，应执行：

1. 校验 `run_id`、`run_node_id`、`agent_task_id`
2. 写入 `AgentTaskReceipt`
3. 更新 `AgentTask`
4. 追加节点日志
5. 更新节点状态
6. 必要时写入附件/导出物记录
7. 触发流程推进或人工确认

## 11. 产品交互建议

### 11.1 发起入口

新增两种发起方式：

1. 从固定模板发起
2. 让龙虾生成流程草案

这样可以兼容现有固定模板体系，不必一次性替换。

### 11.2 节点详情页

节点详情页建议突出三个事实：

* 谁执行了
* 产出了什么
* 谁对结果负责

建议展示区块：

* 节点目标
* 执行主体
* 结果责任人
* 输入信息
* 执行日志
* 结构化结果
* 附件/导出物
* 确认动作
* 异常升级入口

### 11.3 运行看板

看板里建议新增对龙虾节点的可视化状态：

* 执行中
* 待确认
* 异常阻塞
* 自动完成率
* 人工确认耗时

### 11.4 交付页

交付页建议不只展示结果，还要展示：

* 最终结果由哪个龙虾产出
* 哪些节点由人确认
* 最终由谁签收

这样更能体现“结果责任链”。

## 12. MVP 实现边界

### 12.1 本轮建议优先实现

第一阶段建议先做：

1. 自然语言生成流程草案
2. 流程草案确认后创建流程实例
3. 自动节点触发真实龙虾执行
4. 龙虾回执接口
5. 自动节点执行状态和结果展示
6. 人工确认节点承接龙虾结果

### 12.2 本轮建议暂不做

为控制复杂度，本轮建议不做：

* 并行节点
* 条件分支
* 回退编排器
* 通用模板设计器
* 多龙虾协作编排
* 自主长链决策
* 完整通知中心
* 完整权限矩阵
* 多租户隔离

### 12.3 自动节点任务类型建议收敛

MVP 建议先只支持 3 类龙虾任务：

1. 数据查询 / 汇总
2. 批量操作 / 批量绑定
3. 结果导出 / 对账

这三类最容易验证“龙虾执行，人确认负责”的闭环。

### 13.1 前端改造建议

前端建议新增：

* 流程草案创建页
* 流程草案确认页
* 节点执行态卡片
* 节点结构化结果卡片

### 13.2 后端改造建议

后端建议新增：

* `FlowDraft` 相关接口
* `AgentTask` 创建和查询接口
* 龙虾回执接口
* 自动节点触发器
* 节点执行结果落库逻辑
* 节点流转规则增强

### 13.3 OpenClaw / 龙虾集成建议

龙虾侧建议提供两类能力：

1. 编排能力
   * 接收自然语言需求
   * 输出结构化流程草案
2. 执行能力
   * 接收节点任务
   * 执行工具调用或操作
   * 回传标准回执

## 14. 成功标准

当以下链路跑通时，龙虾集成方案可以视为成立：

1. 发起人输入自然语言需求
2. 龙虾生成流程草案
3. 人确认草案并创建流程
4. 流程流转到自动节点
5. 龙虾被真实触发执行
6. 龙虾回传结构化回执
7. 流程根据回执自动流转或进入人工确认
8. 最终签收人确认结果并完成交付

如果这条主链路稳定成立，虾线就不再只是“流程展示平台”，而会真正成为：

**连接龙虾生产力与人工责任闭环的结果型协作系统。**
