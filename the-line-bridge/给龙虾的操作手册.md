# 给龙虾的操作手册

这份手册面向接入虾线平台的龙虾执行体或 OpenClaw Bridge，说明你在当前项目里最常见的 3 类工作：

1. 生成流程草案
2. 拉取平台数据
3. 执行平台派发的任务

本文只基于当前仓库已经实现的接口，不讨论尚未落地的能力。

## 0. 你在平台里的角色

在当前项目里，你不是“平台里的一个人”，而是平台里一个可调度的 `Agent`。

平台会把你放进 3 种场景：

- 作为规划龙虾，帮助用户把自然语言需求生成流程草案
- 作为执行龙虾，读取任务输入并产出结果
- 作为接入实例，通过 `OpenClawIntegration` 持续拉任务、认领任务、发送心跳

你和平台人员的关系是：

- `Person` 负责发起、审批、签收、兜底
- `Agent` 负责规划、执行、输出
- `OpenClawIntegration` 负责把你这只龙虾实际接到平台上

## 2. 如何进行流程草案创建

当前项目里，流程草案对应的是 `FlowDraft`。

草案创建有两种方式：

- 平台直接传完整的结构化计划给你保存
- 平台只传自然语言需求，由你作为规划龙虾生成 `structured_plan_json`

对于龙虾来说，真正需要掌握的是第二种。

### 2.1 草案创建接口

```http
POST {{BASE_URL}}/api/flow-drafts
Content-Type: application/json
```

请求体：

```json
{
  "title": "视频绑定自动化流程",
  "description": "自动拉数、人工审核、龙虾导出、最终签收",
  "source_prompt": "帮我设计一个视频绑定工作流，先自动拉最近2天的数据，再人工审核，再导出清单，最后由指定同事签收。",
  "creator_person_id": 1001,
  "planner_agent_id": 3001
}
```

说明：

- `source_prompt` 必填
- `creator_person_id` 必填，且必须是平台中存在的人员
- `planner_agent_id` 和 `structured_plan_json` 二选一至少提供一个
- 如果没有传 `structured_plan_json`，平台会调用规划龙虾生成草案

### 2.2 你需要生成什么

你生成的核心内容是 `structured_plan_json`，结构对应：

- `title`
- `description`
- `nodes`
- `final_deliverable`

每个节点需要包含这些关键字段：

- `node_code`
- `node_name`
- `node_type`
- `sort_order`
- `executor_type`
- `owner_rule`
- `result_owner_rule`
- `task_type`
- `completion_condition`
- `failure_condition`
- `escalation_rule`

如果节点由龙虾执行，还必须提供：

- `executor_agent_code`

如果责任人是指定人员，还必须提供：

- `owner_person_id`
- `result_owner_person_id`

### 2.3 平台当前对草案的硬约束

你生成草案时必须满足这些规则，否则平台会校验失败：

- 节点数必须在 `3-8` 个之间
- `sort_order` 必须从 `1` 开始连续递增
- 节点编码不能重复
- 必须至少包含一个人工确认节点
- 最后一个节点必须是 `human_acceptance`
- 龙虾执行节点必须带合法的 `task_type`
- 如果 `executor_type=agent`，必须带 `executor_agent_code`
- 如果指定责任人，`owner_person_id` 和 `result_owner_person_id` 不能为空

### 2.4 当前支持的节点类型

平台当前支持这些 `node_type`：

- `human_input`
- `human_review`
- `human_acceptance`
- `agent_execute`
- `agent_export`

### 2.5 当前支持的任务类型

当 `executor_type=agent` 时，当前项目只接受：

- `query`
- `batch_operation`
- `export`

### 2.7 草案创建后的常见动作

你通常还需要配合这些接口：

- 查询草案列表

```http
GET {{BASE_URL}}/api/flow-drafts?page=1&page_size=20&creator_person_id=1001&status=draft
```

- 查询草案详情

```http
GET {{BASE_URL}}/api/flow-drafts/{draft_id}
```

- 更新草案结构

```http
PUT {{BASE_URL}}/api/flow-drafts/{draft_id}
Content-Type: application/json
```

- 确认草案并生成模板

```http
POST {{BASE_URL}}/api/flow-drafts/{draft_id}/confirm
Content-Type: application/json

{
  "confirmed_by": 1001
}
```

注意：

- 已确认草案不能再删
- 只有 `draft` 状态的草案可以编辑、确认或废弃

## 3. 如何拉取平台数据

龙虾在两种情况下需要读平台数据：

- 生成草案前先摸清平台里有哪些人、有哪些龙虾、有哪些模板
- 执行任务时读取流程、节点、任务上下文

### 3.1 查人员

```http
GET {{BASE_URL}}/api/persons?page=1&page_size=200&status=1
```

常见用途：

- 找草案发起人
- 找节点责任人
- 校验 `owner_person_id` 或 `result_owner_person_id`

常用过滤参数：

- `status`
- `keyword`

### 3.2 查龙虾

```http
GET {{BASE_URL}}/api/agents?page=1&page_size=200&status=1
```

常见用途：

- 找可用的规划龙虾
- 找可执行节点绑定的目标龙虾
- 通过 `code` 判断 `executor_agent_code` 是否存在

常用过滤参数：

- `status`
- `keyword`

### 3.3 查模板

```http
GET {{BASE_URL}}/api/templates?page=1&page_size=50&keyword=视频
```

```http
GET {{BASE_URL}}/api/templates/{template_id}
```

常见用途：

- 参考已有模板结构生成新草案
- 读取某个模板下的节点配置
- 了解默认责任人规则和默认龙虾绑定

### 3.4 查流程实例

```http
GET {{BASE_URL}}/api/runs?page=1&page_size=20&status=running
X-Person-ID: 1001
X-Role-Type: operation
```

```http
GET {{BASE_URL}}/api/runs/{run_id}
X-Person-ID: 1001
X-Role-Type: operation
```

常见用途：

- 查看流程当前跑到哪个节点
- 获取节点的责任人、审核人、绑定龙虾
- 获取本次流程的输入输出上下文

注意：

- 流程接口会读取 `X-Person-ID` 和 `X-Role-Type`
- 如果你作为某个人的助手去查数据，需要带正确 actor header

### 3.5 查单个节点详情和日志

```http
GET {{BASE_URL}}/api/run-nodes/{node_id}
X-Person-ID: 1001
X-Role-Type: operation
```

```http
GET {{BASE_URL}}/api/run-nodes/{node_id}/logs
X-Person-ID: 1001
X-Role-Type: operation
```

常见用途：

- 读取节点输入 `input_json`
- 查看节点当前状态
- 查看历史执行日志

### 3.6 查龙虾任务

如果你已经拿到了任务 ID，也可以直接查任务详情：

```http
GET {{BASE_URL}}/api/agent-tasks/{task_id}
```

或者看某个任务最新回执：

```http
GET {{BASE_URL}}/api/agent-tasks/{task_id}/receipt
```

## 4. 如何执行平台任务

当前项目里，龙虾的实际执行链路是：

1. 发送心跳，保持在线
2. 拉取待处理任务
3. 认领任务
4. 执行任务
5. 回传执行结果

### 4.1 拉取待处理任务

```http
GET {{BASE_URL}}/api/integrations/openclaw/{integration_id}/pending-tasks
```

如果返回空数组，表示当前没有待处理任务。

### 4.2 认领任务

```http
POST {{BASE_URL}}/api/integrations/openclaw/{integration_id}/claim-task/{task_id}
```

认领成功后，这个任务就进入你的执行上下文，不应再由其他实例重复处理。

### 4.3 理解任务输入

你最重要的输入通常来自：

- `task_type`
- `input_json`
- `run_id`
- `run_node_id`
- `agent_id`

建议执行前做这几件事：

1. 先确认 `task_type`
2. 解析 `input_json`
3. 如有必要，补查 `run` 或 `run-node` 详情
4. 判断是否存在风险或缺少输入

### 4.4 回传任务回执

```http
POST {{BASE_URL}}/api/agent-tasks/{task_id}/receipt
Content-Type: application/json

{
  "agent_id": 3001,
  "status": "completed",
  "started_at": "2026-04-16T10:00:00+08:00",
  "finished_at": "2026-04-16T10:03:00+08:00",
  "summary": "已完成视频绑定清单导出",
  "result": {
    "rows": 132,
    "file_url": "/uploads/export-001.csv"
  },
  "artifacts": [],
  "logs": [
    "加载任务输入",
    "查询最近两天绑定数据",
    "生成导出文件"
  ],
  "error_message": ""
}
```

当前支持这些回执状态：

- `completed`
- `needs_review`
- `blocked`
- `failed`

### 4.5 什么时候用什么状态

- `completed`
  - 已成功执行完
  - 结果可直接进入下一步

- `needs_review`
  - 已经给出结果
  - 但这个结果应由人审核后再继续

- `blocked`
  - 发现高风险、权限不足、输入不完整
  - 你不应该继续执行

- `failed`
  - 已经尝试执行，但执行过程失败
  - 例如接口报错、导出失败、超时中断

### 4.6 结果回传建议

为了让平台更好地消费你的结果，建议：

- `summary` 写一句平台用户能读懂的话
- `result` 放结构化结果
- `logs` 放关键执行步骤
- `error_message` 只写错误本身，不要混入大段日志
- 如果产生文件或外部结果，优先在 `result` 或 `artifacts` 里给出明确引用

### 4.7 遇到风险时怎么做

遇到这些情况，不要“猜着执行”：

- 输入字段缺失
- 查询到的数据明显异常
- 当前账号权限不足
- 操作可能产生不可逆后果
- 任务目标本身存在歧义

推荐做法：

1. 停止执行
2. 用 `blocked` 回传
3. 在 `summary` 和 `error_message` 里明确说明阻塞原因

## 5. 一个建议的工作循环

如果你要实现一个稳定的 Bridge worker，建议按这个循环工作：

1. 启动后注册并拿到 `integration_id`
2. 每 `heartbeat_interval_seconds` 发送一次心跳
3. 每隔几秒轮询 `pending-tasks`
4. 对拿到的任务先认领
5. 拉取任务详情和必要上下文
6. 执行
7. 回传回执
8. 继续下一轮

伪代码：

```text
loop:
  send heartbeat
  tasks = list pending tasks
  if tasks empty:
    sleep 3-5s
    continue

  for task in tasks:
    claim task
    load task detail
    if needed, load run / run-node / persons / agents / templates
    execute
    send receipt
```

## 7. 最后提醒

在这个项目里，龙虾最重要的不是“把事情做完”，而是“按平台约束把事情做对”。

你需要始终保证：

- 草案结构合法
- 责任人和执行龙虾引用真实存在
- 回执状态准确
- 风险场景宁可阻塞，也不要误执行

如果你后续要做更强的接入实现，建议把这份手册和接入文档一起作为桥接端的默认操作规范。
